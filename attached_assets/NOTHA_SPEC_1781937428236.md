# NOTHA — Especificação de Arquitetura
**Agente de negociação de produtos físicos via WhatsApp**

Versão 1.0 — Documento de referência para implementação de backend e banco de dados.

---

## 1. Visão geral

O NOTHA é um agente de inteligência artificial que intermedia a compra e venda de produtos físicos via WhatsApp. Diferente de um corretor tradicional que apenas aproxima as partes, o NOTHA:

1. Avalia e precifica produtos no momento do cadastro
2. Conduz negociação entre comprador e vendedor através de agentes representantes (proxies) que disputam o melhor valor para cada lado
3. Submete o acordo alcançado para confirmação humana de ambas as partes
4. Coordena a logística de entrega ou retirada do produto
5. Processa o pagamento via Asaas, retendo o valor de forma controlada pelo próprio backend até a confirmação de entrega — sem exigir conta própria no Asaas (e portanto sem KYC bancário) de vendedor, comprador ou entregador, que recebem via chave Pix de qualquer banco

A empresa operadora atual é a **MAISOR CAPITAL LTDA** (CNPJ ativo), com o produto NOTHA rodando sob esse CNPJ até a eventual constituição de uma entidade própria (NOTHA LTDA).

### 1.1 Princípio arquitetural central

> **Conversação é responsabilidade de LLM. Decisão sobre dinheiro é responsabilidade de código determinístico, com LLM operando apenas dentro de limites (guard rails) fixados por humanos.**

Esse princípio orienta toda a separação de agentes descrita neste documento. LLMs nunca têm autoridade final sobre valores monetários — eles propõem, argumentam e sintetizam, mas todo valor é validado contra limites declarados pelos humanos antes de qualquer execução financeira.

## 2. Os agentes

### 2.1 Conversation Agent

**Responsabilidade:** única interface de linguagem natural com humanos (comprador e vendedor).

**O que faz:**
- Interpreta mensagens em linguagem natural e extrai intenção estruturada (JSON)
- Traduz decisões estruturadas do sistema em respostas naturais
- Mantém tom humano e contextual durante toda a interação

**O que NÃO faz:**
- Não decide preços
- Não acessa diretamente o Asaas
- Não tem memória própria — todo estado vem do State Store

**Parametrização:** roda com `role: seller` ou `role: buyer`. Mesmo "molde" de agente, prompt parametrizado pelo papel e pelos dados da negociação ativa.

**Exemplo de extração:**
```json
// Input: "acho que consigo pagar uns 200 e pouco"
{
  "intencao": "contraproposta",
  "valor_estimado": 200,
  "confianca": "media",
  "contexto_extra": null
}
```

---

### 2.2 Pricing/Appraisal Agent

**Responsabilidade:** formar uma sugestão de preço (e preço mínimo) no momento do cadastro do produto. Roda **uma única vez por listing**, de forma assíncrona — não está sob pressão de tempo de conversa.

**Fontes de evidência consultadas:**

| Fonte | O que traz | Como é obtido |
|---|---|---|
| Mercado externo | Preço de produto novo/similar | Web search |
| Histórico interno | Preço final de negociações similares já fechadas | Query SQL em `negotiations` + `listings` |
| Demanda observada | Quantidade e valor de interesses anteriores em produtos similares | Query SQL em `interest_queue` |
| Avaliação visual | Estado de conservação, autenticidade aparente | Modelo de visão sobre as fotos enviadas |
| Descrição textual | Detalhes que o vendedor informou (uso, defeitos, comprovantes) | Texto do próprio listing |

**Saída estruturada:**
```json
{
  "preco_sugerido": 230,
  "preco_minimo_sugerido": 195,
  "justificativa": "Baseado em 8 vendas similares (média R$225) e estado de conservação bom identificado nas fotos.",
  "confianca": "alta",
  "fontes": ["historico_interno", "avaliacao_visual", "mercado_externo"]
}
```

**Comportamento com o preço informado pelo vendedor:**
- Se o vendedor não informou preço → o agente sugere `preco_sugerido` e `preco_minimo_sugerido`, vendedor confirma ou ajusta
- Se informou e a diferença para a sugestão for >20% → o agente alerta e sugere ajuste, mas não força
- Se informou e está dentro da faixa esperada → confirmação silenciosa

**Regra inviolável:** o preço final em `listings.preco_minimo` só é gravado após confirmação humana do vendedor. O agente nunca publica um listing sem essa confirmação.

### 2.3 Negotiation Engine

**Responsabilidade:** decidir se uma oferta é aceita, recusada ou gera contraproposta — via regra determinística, não LLM — para o modo de negociação direta (sem proxies).

**Por que não é LLM:** decisão monetária precisa ser auditável e repetível. Mesma entrada deve sempre produzir a mesma saída.

**Lógica base:**
```python
def decidir_oferta(oferta, preco_minimo, contexto_extra=None):
    if oferta >= preco_minimo:
        return "aceitar"
    if oferta >= preco_minimo * 0.9 and not contexto_extra:
        return "contrapropor"
    if contexto_extra:
        avaliacao = contextual_evaluator_agent.avaliar(oferta, preco_minimo, contexto_extra)
        ajuste_real = max(avaliacao.ajuste_permitido, preco_minimo * AJUSTE_MAXIMO_PERMITIDO)
        if oferta >= preco_minimo + ajuste_real:
            return "aceitar_com_justificativa"
    return "recusar"
```

**Contextual Evaluator** (sub-componente, é LLM): avalia fatores não-numéricos que podem justificar desconto (retirada imediata, pagamento à vista, compra em volume). Retorna uma *sugestão* de ajuste — nunca decide sozinho. O ajuste sempre passa por um teto fixo em código (`AJUSTE_MAXIMO_PERMITIDO`, ex: nunca mais que 15% abaixo do mínimo).

### 2.4 Buyer Proxy Agent e Seller Proxy Agent

**Responsabilidade:** modelo principal de negociação do NOTHA. Cada agente representa um lado e busca o melhor valor possível para ele, dentro do limite que esse lado declarou.

**Entrada de cada proxy:**
```json
{
  "limite": { "minimo_ou_maximo": 190, "ideal": 240 },
  "dados_produto": { /* avaliação do Pricing Agent, histórico, estado */ },
  "historico_rodada": [ /* propostas e argumentos anteriores */ ],
  "valores_rejeitados_pela_contraparte_humana": [ /* sinal de rodadas anteriores */ ]
}
```

**Saída de cada rodada:**
```json
{
  "decisao": "aceitar | contrapropor",
  "valor": 215,
  "argumento": "Este modelo vendeu em média por R$230 nas últimas 8 negociações similares, com estado de conservação muito bom confirmado nas fotos."
}
```

**Guard rail obrigatório (código, não LLM):**
```python
def validar_resposta_proxy(resposta, limite, eh_vendedor):
    if eh_vendedor and resposta.valor < limite.minimo:
        raise ValorForaDosLimites("vendedor não pode aceitar abaixo do mínimo declarado")
    if not eh_vendedor and resposta.valor > limite.maximo:
        raise ValorForaDosLimites("comprador não pode oferecer acima do máximo declarado")
```

Nenhuma saída de proxy é executada sem passar por essa validação. Isso é não-negociável na implementação.

**Limite de rodadas internas:** `MAX_RODADAS_PROXY = 6` (configurável). Se não convergirem, cai no fallback de interseção matemática:
```python
def fallback_interseccao(comprador_max, vendedor_min):
    if comprador_max < vendedor_min:
        return None  # sem acordo possível
    return (comprador_max + vendedor_min) / 2
```

### 2.5 Transaction Agent

**Responsabilidade:** execução financeira via Asaas. Camada de engenharia tradicional (retry, idempotência, tratamento de exceção) — não é um agente "de IA" no sentido criativo.

**Modelo escolhido: retenção controlada pelo backend, sem subconta/KYC de terceiros.**

Avaliamos dois modelos do Asaas:

| Modelo | Onboarding | Retenção condicional | Decisão |
|---|---|---|---|
| Conta Escrow + subcontas | Exige KYC completo de vendedor/comprador/entregador | Nativa do Asaas | ❌ Descartado — fricção incompatível com o produto |
| Transferência para chave Pix externa | Zero onboarding bancário — qualquer chave Pix de qualquer banco | Controlada pelo próprio backend do NOTHA | ✅ Adotado |

No modelo adotado, **apenas a conta principal da MAISOR CAPITAL** tem KYC no Asaas. Vendedor, comprador e entregador apenas informam uma chave Pix já existente (de qualquer banco) para receber valores — nenhum deles precisa de conta própria no Asaas.

**Implicação operacional:** o dinheiro do comprador entra na cobrança Pix gerada na conta principal da MAISOR CAPITAL e **permanece ali, retido**, até a confirmação de entrega/retirada. A liberação é uma ação explícita do backend (não automática pelo Asaas), via chamada a transferência para chave Pix externa.

**Operações:**
- Criar cobrança para o comprador (valor do produto + entrega, se aplicável)
- Escutar webhook de confirmação de pagamento
- Manter o valor retido (`status_retencao = 'retido_aguardando_entrega'`) até confirmação mútua de entrega
- Disparar transferência via chave Pix ao vendedor (e ao entregador, se houver) somente após confirmação
- Disparar estorno automático para a origem, caso o prazo final de resolução se esgote sem retirada/entrega concluída
- Tratar falhas (timeout do Asaas, pagamento não confirmado, cancelamento)

**Regra de negócio chave:** como o valor fica temporariamente na conta principal da MAISOR CAPITAL (e não em subconta do vendedor), o NOTHA atua como intermediário financeiro que retém temporariamente valores de terceiros até confirmação de entrega — isso deve estar documentado de forma explícita nos Termos de Uso do produto.

**Controle de conciliação obrigatório:** como o saldo agregado da conta Asaas não distingue, por si só, quanto pertence a qual transação pendente, o backend deve manter uma visão de saldo retido por transação e validar periodicamente que a soma de tudo retido (`SUM(valor_total) WHERE status_retencao = 'retido_aguardando_entrega'`) é compatível com o saldo real na conta Asaas. Divergência aqui é sinal de bug de conciliação e deve gerar alerta.

---

### 2.6 Logistics/Delivery Agent

**Responsabilidade:** decidir e coordenar a modalidade de entrega após o acordo de compra/venda ser confirmado pelos humanos, e antes da liberação do valor retido.

**Fluxo de decisão:**
1. Pergunta ao comprador: retirar o produto ou contratar entrega de confiança do NOTHA?
2. Se **retirada**: solicita data e horário de retirada, cria agendamento, notifica o vendedor. Cobrança gerada é só o valor do produto.
3. Se **entrega**: aciona o **Delivery Proxy** para encontrar e negociar com um entregador disponível na região (mesmo padrão de proxy com guard rails que regula Buyer/Seller Proxy). Cobrança gerada já inclui produto + entrega, em valor único.

**Delivery Proxy Agent:** estrutura análoga ao Buyer/Seller Proxy — o entregador declara um valor mínimo aceitável pelo trecho, o sistema negocia dentro do limite que o comprador/vendedor topam pagar pela entrega, com o mesmo guard rail de validação em código antes de qualquer confirmação.

**Acompanhamento pós-agendamento (retirada):**
- Se o comprador confirma a retirada no prazo combinado → vendedor confirma a entrega física → valor é liberado.
- Se o prazo agendado vence sem confirmação → o agente pergunta ao comprador se quer converter para entrega NOTHA.
  - Se aceitar → gera cobrança **adicional** apenas do valor da entrega (produto já está pago) e segue o fluxo de entrega.
  - Se recusar ou não responder → ver seção 4.4 (relistagem e estorno).

---

### 2.7 Orquestrador

**Responsabilidade:** roteamento. Recebe toda mensagem entrante, consulta o State Store, decide qual agente aciona, escreve o resultado de volta.

**Princípio de design:** o Orquestrador deve ter o mínimo de lógica de negócio possível. Toda decisão sobre preço vive na Negotiation Engine ou nos Proxies — o Orquestrador apenas direciona o fluxo.

---

### 2.8 State Store

Não é um agente — é a memória persistente compartilhada por todos os agentes. PostgreSQL. Nenhum agente mantém contexto "na cabeça"; tudo é lido e escrito aqui, permitindo que qualquer agente reinicie sem perda de estado.

## 3. Modelo de dados

### 3.1 Identidade do usuário: CPF como âncora estável, telefone como canal

Princípio de design: **fricção mínima de onboarding**, mas com identidade resiliente a troca de número de WhatsApp — situação comum e recorrente (linhas pré-pagas, perda de chip, etc.).

**Decisão de modelagem:** o telefone do WhatsApp é o canal de entrada em toda mensagem, mas **não é a chave de identidade**. Quem identifica a pessoa de fato é o CPF. Isso evita que uma troca de número crie um usuário duplicado, perdendo histórico de listings, negociações e reputação.

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    cpf VARCHAR(14) UNIQUE,               -- identidade estável da pessoa
    nome VARCHAR(200),
    created_at TIMESTAMP DEFAULT now(),
    updated_at TIMESTAMP DEFAULT now()
);

CREATE TABLE user_phone_numbers (
    id SERIAL PRIMARY KEY,
    user_id INT NOT NULL REFERENCES users(id),
    telefone VARCHAR(20) UNIQUE NOT NULL,
    ativo BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT now()
);

-- garante que cada usuário tenha apenas um número ativo por vez
CREATE UNIQUE INDEX idx_one_active_phone_per_user
    ON user_phone_numbers(user_id) WHERE ativo = TRUE;
```

**Reconhecimento de número novo (possível troca de telefone):**

```python
async def identificar_usuario(telefone_recebido):
    vinculo = await db.fetch_one(
        "SELECT user_id FROM user_phone_numbers WHERE telefone=:tel AND ativo=TRUE",
        {"tel": telefone_recebido}
    )
    if vinculo:
        return vinculo.user_id  # caso normal, número já conhecido
    return None  # número nunca visto -- aciona fluxo de recuperação/novo cadastro


async def tratar_numero_desconhecido(telefone_novo):
    resposta = await conversation_agent.perguntar(
        telefone_novo,
        "Antes de continuar: você já usou o NOTHA com outro número? "
        "Se sim, me informa seu CPF para eu recuperar seu histórico."
    )
    if resposta.tem_cpf:
        usuario_existente = await db.fetch_one(
            "SELECT * FROM users WHERE cpf=:cpf", {"cpf": resposta.cpf}
        )
        if usuario_existente:
            return await migrar_numero(usuario_existente.id, telefone_novo)
    return await criar_usuario_novo(telefone_novo)


async def migrar_numero(user_id, telefone_novo):
    async with db.transaction():
        await db.execute(
            "UPDATE user_phone_numbers SET ativo=FALSE WHERE user_id=:id",
            {"id": user_id}
        )
        await db.execute("""
            INSERT INTO user_phone_numbers (user_id, telefone, ativo)
            VALUES (:user_id, :telefone, TRUE)
            ON CONFLICT (telefone) DO UPDATE SET user_id=:user_id, ativo=TRUE
        """, {"user_id": user_id, "telefone": telefone_novo})
    return user_id
```

**Por que o risco aqui é baixo:** a plataforma nunca expõe dados pessoais de um usuário a outro — apenas dados de produtos (públicos por natureza, é um catálogo). Mesmo que alguém tentasse se apresentar com um CPF que não é seu para recuperar uma conta, o que ganharia é acesso ao histórico de listings/negociações de outra pessoa dentro da conversa — não dinheiro nem dados bancários de terceiros, já que a chave Pix de cada um só é usada internamente pelo backend para transferência, nunca exibida a outros usuários.

### 3.2 Perfis por papel (vendedor, comprador, entregador)

Dados são coletados *just-in-time*, apenas quando a ação específica exige — nunca todos de uma vez no primeiro contato.

| Nível | Quando é coletado | Dados |
|---|---|---|
| 1 — Identificação mínima | Qualquer interação (buscar produto, iniciar conversa) | Nome, CPF (telefone já vem do WhatsApp) |
| 2 — Perfil vendedor | Ao listar um produto | Endereço de retirada, horários disponíveis, chave Pix |
| 3 — Perfil comprador | Ao fechar uma compra com entrega (não ao só buscar) | Endereço de entrega |
| 4 — Perfil entregador | Ao aceitar realizar uma entrega (papel opcional, qualquer usuário pode assumir) | Chave Pix, região de atuação |

```sql
CREATE TABLE seller_profile (
    user_id INT PRIMARY KEY REFERENCES users(id),
    endereco_retirada TEXT NOT NULL,
    horarios_disponiveis JSONB,           -- ex: [{"dia": "seg-sex", "periodo": "18h-21h"}]
    chave_pix VARCHAR(150) NOT NULL,
    chave_pix_titular_confirmado VARCHAR(200),  -- nome retornado pelo Asaas na validação
    created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE buyer_profile (
    user_id INT PRIMARY KEY REFERENCES users(id),
    endereco_entrega TEXT,
    created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE courier_profile (
    user_id INT PRIMARY KEY REFERENCES users(id),
    chave_pix VARCHAR(150) NOT NULL,
    chave_pix_titular_confirmado VARCHAR(200),
    regiao_atuacao TEXT,
    created_at TIMESTAMP DEFAULT now()
);
```

**Por que tabelas separadas por papel, e não um único `users` com colunas opcionais:** um mesmo `user_id` acumula papéis diferentes ao longo do tempo (compra hoje, vende depois, eventualmente faz uma entrega). Separar evita uma tabela central poluída com colunas majoritariamente `NULL`.

**Validação de CPF:** apenas formato (dígitos verificadores), não existência documental — proporcional ao risco do modelo, que não envolve custódia bancária de terceiros, apenas retenção temporária e curta até confirmação de entrega.

**Validação de chave Pix — aproveitando infraestrutura nacional já existente:** o Banco Central garante estruturalmente que uma chave Pix do tipo CPF só pode pertencer à conta cujo titular tem aquele CPF — não é possível cadastrar CPF de terceiro como chave própria, o próprio banco preenche o CPF automaticamente sem permitir alteração. Isso significa que o NOTHA não precisa reconstruir essa verificação — apenas consumir o que o Asaas já retorna ao consultar a chave:

```python
async def validar_chave_pix_antes_de_salvar(chave_pix):
    info_titular = await asaas_client.consultar_chave_pix(chave_pix)
    if not info_titular:
        return {"valido": False, "motivo": "chave_pix_invalida_ou_nao_encontrada"}
    return {"valido": True, "nome_titular": info_titular.nome}


async def cadastrar_chave_pix(user_id, chave_pix, tabela_perfil):
    validacao = await validar_chave_pix_antes_de_salvar(chave_pix)
    if not validacao["valido"]:
        return {"erro": "Não consegui validar essa chave Pix, pode confirmar?"}

    confirmou = await conversation_agent.confirmar(
        user_id,
        f"Confirmo que a chave Pix pertence a {validacao['nome_titular']}?"
    )
    if confirmou:
        await db.execute(f"""
            UPDATE {tabela_perfil}
            SET chave_pix=:chave, chave_pix_titular_confirmado=:nome
            WHERE user_id=:id
        """, {"chave": chave_pix, "nome": validacao["nome_titular"], "id": user_id})
```

Exibir o nome do titular antes de confirmar pega erros de digitação na hora e torna visível qualquer tentativa de informar uma chave que não é do próprio usuário.

**Nota de conformidade:** CPF, endereço e chave Pix são dados pessoais sob a LGPD. O fluxo de coleta deve linkar a política de privacidade do NOTHA na primeira coleta de dado sensível. Recomenda-se revisão jurídica formal antes do lançamento em produção.

**Verificação documental — reservada a casos excepcionais:** o fluxo normal de uso (listar, negociar, comprar) nunca exige documento. Solicitação de documento de identidade + foto segurando o documento é reservada a casos pontuais e de maior risco: disputa de transação, exportação de dados pessoais (direito LGPD), ou recuperação de conta quando a verificação por CPF (seção 3.1) for considerada insuficiente pelo time de operações.

**Lógica de verificação de dados pendentes (usada pelo Orquestrador antes de cada ação):**

```python
async def verificar_dados_necessarios(user_id, acao):
    user = await buscar_usuario(user_id)

    if not user.nome or not user.cpf:
        return {"falta": ["nome", "cpf"], "motivo": "identificacao_minima"}

    if acao == "listar_produto":
        seller = await buscar_seller_profile(user_id)
        faltantes = [campo for campo in ["endereco_retirada", "horarios_disponiveis", "chave_pix"]
                     if not seller or not getattr(seller, campo, None)]
        if faltantes:
            return {"falta": faltantes, "motivo": "perfil_vendedor_incompleto"}

    if acao == "buscar_produto":
        return {"falta": [], "motivo": None}  # nível 1 já é suficiente

    return {"falta": [], "motivo": None}
```

### 3.3 `listings` — catálogo de produtos

```sql
CREATE TABLE listings (
    id SERIAL PRIMARY KEY,
    seller_id INT NOT NULL,
    descricao TEXT NOT NULL,
    categoria VARCHAR(100),
    fotos JSONB,                          -- array de URLs
    preco_informado_vendedor NUMERIC,     -- pode ser NULL
    preco_sugerido NUMERIC,               -- saída do Pricing Agent
    preco_anunciado NUMERIC NOT NULL,     -- valor final confirmado, exibido publicamente
    preco_minimo NUMERIC NOT NULL,        -- nunca exposto ao comprador
    appraisal_data JSONB,                 -- justificativa e fontes do Pricing Agent
    status VARCHAR(30) NOT NULL DEFAULT 'disponivel',
        -- disponivel | em_negociacao | vendido | cancelado
    created_at TIMESTAMP DEFAULT now(),
    updated_at TIMESTAMP DEFAULT now()
);

CREATE INDEX idx_listings_status ON listings(status);
CREATE INDEX idx_listings_categoria ON listings(categoria);
```

### 3.4 `negotiations` — negociação ativa por listing

```sql
CREATE TABLE negotiations (
    id SERIAL PRIMARY KEY,
    listing_id INT NOT NULL REFERENCES listings(id),
    buyer_id INT NOT NULL,
    modo VARCHAR(20) NOT NULL DEFAULT 'proxy',  -- proxy | direta
    status VARCHAR(30) NOT NULL DEFAULT 'ativa',
        -- ativa | aceita | aguardando_pagamento | paga
        -- recusada | expirada | expirada_por_timeout | cancelada_pelo_usuario | sem_acordo
    preco_atual_proposto NUMERIC,
    limite_comprador JSONB,               -- {"maximo": 220, "ideal": 180}
    limite_vendedor JSONB,                -- {"minimo": 190, "ideal": 240}
    responder_until TIMESTAMP,            -- prazo da rodada atual (timeout)
    expires_at TIMESTAMP,                 -- prazo total da negociação
    tentativas_humanas INT DEFAULT 0,     -- quantas vezes a proposta foi levada aos humanos
    created_at TIMESTAMP DEFAULT now()
);

CREATE INDEX idx_negotiations_status ON negotiations(status);
CREATE INDEX idx_negotiations_listing ON negotiations(listing_id);
CREATE INDEX idx_negotiations_responder_until ON negotiations(responder_until)
    WHERE status = 'ativa';
```

### 3.7 `interest_queue` — fila de interessados

```sql
CREATE TABLE interest_queue (
    id SERIAL PRIMARY KEY,
    listing_id INT NOT NULL REFERENCES listings(id),
    buyer_id INT NOT NULL,
    oferta_inicial NUMERIC,
    timestamp TIMESTAMP DEFAULT now(),
    UNIQUE(listing_id, buyer_id)
);

CREATE INDEX idx_interest_queue_listing ON interest_queue(listing_id, timestamp);
```

### 3.8 `transactions` — execução financeira

Modelo sem subconta/escrow nativo do Asaas — retenção controlada pelo backend do NOTHA, liberação via transferência para chave Pix externa (ver seção 2.5).

```sql
CREATE TABLE transactions (
    id SERIAL PRIMARY KEY,
    negotiation_id INT NOT NULL REFERENCES negotiations(id),

    -- valores
    valor_produto NUMERIC NOT NULL,
    valor_entrega NUMERIC DEFAULT 0,
    valor_total NUMERIC GENERATED ALWAYS AS (valor_produto + valor_entrega) STORED,
    taxa_notha NUMERIC NOT NULL DEFAULT 0,

    -- modalidade
    modalidade_entrega VARCHAR(20) NOT NULL DEFAULT 'retirada',
        -- retirada | entrega_notha

    -- destinos de repasse (chave Pix de qualquer banco, sem subconta)
    chave_pix_vendedor VARCHAR(150) NOT NULL,
    chave_pix_entregador VARCHAR(150),
    entregador_id INT REFERENCES users(id),

    -- identificadores Asaas
    asaas_charge_id VARCHAR(100),
    asaas_transfer_id_vendedor VARCHAR(100),
    asaas_transfer_id_entregador VARCHAR(100),

    -- status do ciclo de vida
    status VARCHAR(30) NOT NULL DEFAULT 'pendente',
        -- pendente | cobranca_criada | pago | falhou
    status_retencao VARCHAR(30) NOT NULL DEFAULT 'retido_aguardando_entrega',
        -- retido_aguardando_entrega | liberado
        -- retido_aguardando_decisao_pos_falha | estornado_automaticamente | estornado_manualmente

    prazo_estorno_automatico TIMESTAMP,    -- definido apenas quando entra em retido_aguardando_decisao_pos_falha

    created_at TIMESTAMP DEFAULT now(),
    updated_at TIMESTAMP DEFAULT now()
);

CREATE INDEX idx_transactions_status_retencao ON transactions(status_retencao);

-- view de conciliação: soma de tudo retido vs. saldo real esperado na conta Asaas
CREATE VIEW saldo_retido_total AS
SELECT SUM(valor_total) as total_retido
FROM transactions
WHERE status_retencao IN ('retido_aguardando_entrega', 'retido_aguardando_decisao_pos_falha');
```

## 4. Máquina de estados

### 4.1 Estados do `listing`

```
disponivel ──────► em_negociacao ──────► vendido
    ▲                    │
    │                    ▼
    └──────────────  cancelado
```

### 4.2 Estados da `negotiation` (modo proxy)

```
ativa (proxies negociando)
   │
   ▼
[proxies convergem em valor candidato]
   │
   ▼
proposta_ao_vendedor ──► vendedor RECUSA ──► volta para "ativa" (nova rodada, valor marcado como rejeitado)
   │
   ▼ vendedor ACEITA
proposta_ao_comprador ──► comprador RECUSA ──► volta para "ativa" (nova rodada)
   │
   ▼ comprador ACEITA
aceita ──► aguardando_pagamento ──► paga (= listing: vendido)
```

### 4.3 Estados de saída por inatividade/limite

```
ativa ──► [responder_until expirado] ──► expirada_por_timeout
                                              │
                                              ├─► comprador atual volta para o FINAL da fila
                                              └─► próximo da fila é chamado automaticamente

ativa ──► [expires_at total expirado, sem interessados na fila] ──► expirada
                                              │
                                              └─► listing volta a "disponivel"

[tentativas_humanas >= MAX_TENTATIVAS_HUMANAS] ──► sem_acordo
```

### 4.4 Estados de `delivery_confirmations` e liberação financeira

Este fluxo começa depois que `negotiations.status = 'aceita'` e a cobrança é paga (`transactions.status = 'pago'`). É o que conecta a confirmação física de entrega à liberação do valor retido.

```
[pagamento confirmado] ──► Logistics Agent pergunta: retirar ou entrega NOTHA?
        │
        ├──► RETIRADA ESCOLHIDA
        │         │
        │         ▼
        │    Agente solicita data/horário ──► delivery_confirmations: status='agendada'
        │         │
        │         ▼
        │    [chegou prazo_confirmacao]
        │         │
        │         ├──► Comprador confirma retirada + vendedor confirma entrega
        │         │         └──► status='confirmada' ──► LIBERA pagamento (vendedor)
        │         │
        │         └──► Sem confirmação no prazo ──► status='nao_confirmada'
        │                   │
        │                   ▼
        │              Agente pergunta: quer contratar entrega NOTHA agora?
        │                   │
        │                   ├──► Aceita ──► status='convertida_entrega'
        │                   │         └──► gera cobrança ADICIONAL (só valor da entrega)
        │                   │         └──► segue fluxo de ENTREGA NOTHA abaixo
        │                   │
        │                   └──► Recusa ou não responde
        │                             │
        │                             ▼
        │                    listing volta a 'disponivel' (RELISTADO)
        │                    delivery_confirmations.relisted_at = now()
        │                    transactions.status_retencao = 'retido_aguardando_decisao_pos_falha'
        │                    transactions.prazo_estorno_automatico = now() + X dias
        │                             │
        │                             ├──► Comprador responde antes do prazo final
        │                             │         └──► reabre conversa, decide novo caminho
        │                             │
        │                             └──► Prazo final esgotado, sem resposta
        │                                       └──► ESTORNO AUTOMÁTICO para a origem do pagamento
        │                                       └──► status_retencao = 'estornado_automaticamente'
        │
        └──► ENTREGA NOTHA ESCOLHIDA (direto, ou via conversão acima)
                  │
                  ▼
             Delivery Proxy negocia com entregador disponível na região
                  │
                  ▼
             Entregador retira do vendedor, entrega ao comprador
                  │
                  ▼
             Vendedor confirma entrega + Comprador confirma recebimento
                  │
                  ▼
             status='confirmada' ──► LIBERA pagamento (vendedor + entregador)
```

**Ponto de design importante:** a relistagem do produto e a retenção do dinheiro do comprador são desacopladas. O produto pode voltar a circular no catálogo imediatamente (o vendedor não fica bloqueado esperando um comprador que não confirmou), enquanto o valor pago por aquele comprador específico permanece protegido em `transactions` até ele se manifestar ou o prazo de segurança expirar. Se o produto for vendido novamente nesse intervalo, trata-se de uma transação Asaas inteiramente nova e independente — não há tentativa de "recuperar" a venda anterior para o mesmo comprador.

### 5.1 Iniciar negociação (trava o listing)

```python
async def iniciar_negociacao(listing_id, buyer_id, limite_comprador):
    async with db.transaction():
        listing = await db.fetch_one(
            "SELECT * FROM listings WHERE id=:id AND status='disponivel' FOR UPDATE",
            {"id": listing_id}
        )
        if not listing:
            return {"erro": "produto_indisponivel"}

        await db.execute(
            "UPDATE listings SET status='em_negociacao' WHERE id=:id",
            {"id": listing_id}
        )
        negotiation = await db.execute("""
            INSERT INTO negotiations
                (listing_id, buyer_id, modo, status, limite_comprador, responder_until, expires_at)
            VALUES (:listing_id, :buyer_id, 'proxy', 'ativa', :limite, :prazo_rodada, :prazo_total)
            RETURNING *
        """, {
            "listing_id": listing_id,
            "buyer_id": buyer_id,
            "limite": limite_comprador,
            "prazo_rodada": datetime.now() + timedelta(minutes=30),
            "prazo_total": datetime.now() + timedelta(hours=24)
        })
        return negotiation
```

### 5.4 Rodada de negociação entre proxies

```python
async def negociar_entre_proxies(negotiation_id, max_rodadas=6):
    neg = await carregar_negociacao(negotiation_id)
    historico = await carregar_historico_rodadas(negotiation_id)
    oferta_atual = neg.limite_comprador["ideal"]

    for rodada in range(max_rodadas):
        resposta_vendedor = await seller_proxy_agent.avaliar(
            oferta_recebida=oferta_atual,
            limite=neg.limite_vendedor,
            dados_produto=neg.listing.appraisal_data,
            historico=historico
        )
        validar_resposta_proxy(resposta_vendedor, neg.limite_vendedor, eh_vendedor=True)

        if resposta_vendedor.decisao == "aceitar":
            return await propor_aos_humanos(negotiation_id, oferta_atual)

        resposta_comprador = await buyer_proxy_agent.avaliar(
            contraproposta=resposta_vendedor.valor,
            limite=neg.limite_comprador,
            dados_produto=neg.listing.appraisal_data,
            historico=historico
        )
        validar_resposta_proxy(resposta_comprador, neg.limite_comprador, eh_vendedor=False)

        if resposta_comprador.decisao == "aceitar":
            return await propor_aos_humanos(negotiation_id, resposta_vendedor.valor)

        oferta_atual = resposta_comprador.valor
        historico.append(resposta_vendedor)
        historico.append(resposta_comprador)

    # sem convergência -> fallback matemático
    valor_fallback = fallback_interseccao(
        neg.limite_comprador["maximo"], neg.limite_vendedor["minimo"]
    )
    if valor_fallback is None:
        await db.execute(
            "UPDATE negotiations SET status='sem_acordo' WHERE id=:id",
            {"id": negotiation_id}
        )
        return {"status": "sem_acordo"}
    return await propor_aos_humanos(negotiation_id, valor_fallback)
```

### 5.5 Propor aos humanos (sequencial: vendedor → comprador)

```python
async def propor_aos_humanos(negotiation_id, valor):
    neg = await carregar_negociacao(negotiation_id)

    if neg.tentativas_humanas >= MAX_TENTATIVAS_HUMANAS:
        await db.execute(
            "UPDATE negotiations SET status='sem_acordo' WHERE id=:id",
            {"id": negotiation_id}
        )
        return {"status": "sem_acordo"}

    await db.execute("""
        UPDATE negotiations
        SET status='proposta_ao_vendedor', preco_atual_proposto=:valor,
            tentativas_humanas = tentativas_humanas + 1
        WHERE id=:id
    """, {"valor": valor, "id": negotiation_id})

    resposta = await conversation_agent.confirmar(
        neg.listing.seller_id,
        f"Cheguei a um acordo de R${valor} com o comprador. Confirma a venda?"
    )

    if not resposta.aceitou:
        await registrar_rejeicao(negotiation_id, valor, rejeitado_por="vendedor")
        return await negociar_entre_proxies(negotiation_id)  # nova rodada, sabendo do valor rejeitado

    await db.execute(
        "UPDATE negotiations SET status='proposta_ao_comprador' WHERE id=:id",
        {"id": negotiation_id}
    )

    resposta_comprador = await conversation_agent.confirmar(
        neg.buyer_id,
        f"Consegui fechar em R${valor} com o vendedor. Confirma a compra?"
    )

    if not resposta_comprador.aceitou:
        await registrar_rejeicao(negotiation_id, valor, rejeitado_por="comprador")
        return await negociar_entre_proxies(negotiation_id)

    await db.execute(
        "UPDATE negotiations SET status='aceita' WHERE id=:id",
        {"id": negotiation_id}
    )
    return await transaction_agent.iniciar_cobranca(negotiation_id)
```

## 6. Cenário de exemplo completo (sapato)

**Cadastro:**
Vendedor envia foto e descrição de um tênis Nike Air Max, tamanho 42, usado 3x, com caixa. Não informa preço.

→ **Pricing Agent** roda: busca preço de mercado (tênis novo ~R$320), histórico interno (8 vendas similares, média R$225), avaliação visual (estado bom, fator de depreciação 0.75), e sugere:
```json
{"preco_sugerido": 230, "preco_minimo_sugerido": 195}
```
Vendedor confirma. `listings.preco_anunciado = 230`, `listings.preco_minimo = 195`, status `disponivel`.

**Interesse:**
Comprador A demonstra interesse, declara limite (máximo R$220, ideal R$180). `iniciar_negociacao` trava o listing → `em_negociacao`.

Comprador B aparece em seguida → entra em `interest_queue`.

**Negociação entre proxies:**
- Rodada 1: Buyer Proxy abre em R$180. Seller Proxy recusa, contrapropõe R$225 citando histórico de vendas similares.
- Rodada 2: Buyer Proxy oferece R$195, citando preço de mercado para produto usado. Seller Proxy aceita (R$195 ≥ R$195 mínimo).

**Confirmação humana:**
- Sistema propõe R$195 ao vendedor → vendedor confirma.
- Sistema propõe R$195 ao comprador A → comprador A confirma.

**Execução financeira:**
`negotiations.status = 'aceita'` → Transaction Agent cria cobrança no Asaas para Comprador A → webhook confirma pagamento → repasse ao vendedor (R$195 − taxa NOTHA) → `negotiations.status = 'paga'`, `listings.status = 'vendido'`.

Comprador B (na fila) recebe notificação de que o produto foi vendido e é convidado a ver produtos similares.

---

## 7. Cenário de exemplo (timeout e fila)

Mesmo listing, mas o vendedor não responde à proposta de R$195 em 30 minutos (`responder_until` vencido).

→ Job de timeout marca `negotiations.status = 'expirada_por_timeout'`.
→ Comprador A é reinserido no final de `interest_queue` (não descartado).
→ `chamar_proximo_da_fila` identifica Comprador B, inicia nova negociação com ele.
→ Caso B também não chegue a acordo, e a fila se esgote, listing retorna a `disponivel`.
→ Caso o vendedor responda mais tarde, o Comprador A pode ser chamado novamente quando for sua vez na fila — com sua oferta original preservada em `oferta_inicial`.

---

## 8. Cenário de exemplo (retirada não confirmada → conversão para entrega → estorno)

Continuação do cenário do sapato: acordo fechado em R$195, pagamento confirmado, `transactions.status = 'pago'`, `status_retencao = 'retido_aguardando_entrega'`.

**Decisão de logística:**
Logistics Agent pergunta ao Comprador A: retirar ou entrega NOTHA? Comprador escolhe retirada, informa data (sábado) e horário (14h). `delivery_confirmations` criado com `status='agendada'`, vendedor é notificado do agendamento.

**Retirada não confirmada:**
Chega sábado, 14h, e nenhuma das partes confirma a retirada. Job `verificar_retiradas_vencidas` identifica o vencimento, marca `status='nao_confirmada'`.

Agente pergunta ao Comprador A: "Não confirmou a retirada agendada. Quer contratar entrega do NOTHA?"

**Cenário A — comprador aceita entrega:**
`status='convertida_entrega'`. Sistema gera cobrança adicional apenas do valor de entrega (produto já está pago). Delivery Proxy negocia com entregador disponível, fecha em R$15. Comprador paga essa cobrança adicional. Entregador retira do vendedor e entrega ao comprador. Ambos confirmam → `liberar_pagamento_pos_entrega` transfere R$195 (menos taxa) ao vendedor via chave Pix, e R$15 ao entregador via chave Pix.

**Cenário B — comprador não responde:**
Listing volta a `disponivel` (relistado, disponível para nova negociação imediatamente). `transactions.status_retencao = 'retido_aguardando_decisao_pos_falha'`, com `prazo_estorno_automatico = now() + 3 dias`.

- Se o Comprador A responder dentro desses 3 dias, a conversa é reaberta para decidir um novo caminho (nova tentativa de retirada, conversão para entrega, ou solicitação de reembolso).
- Se o prazo se esgotar sem resposta, `verificar_estornos_automaticos` dispara o estorno integral dos R$195 para a origem do pagamento (cartão ou Pix do Comprador A), e `status_retencao = 'estornado_automaticamente'`.

Importante: caso o listing seja vendido novamente para outro comprador nesse intervalo, trata-se de uma transação Asaas totalmente independente — o sistema nunca tenta reconciliar a venda nova com o reembolso pendente do Comprador A.

## 9. Considerações de implementação para o backend

- **Stack:** FastAPI + PostgreSQL (consistente com o restante do ecossistema NOTHA/MAISOR).
- **Jobs periódicos:** usar Celery beat ou similar para `verificar_timeouts`, `verificar_retiradas_vencidas`, `verificar_estornos_automaticos`, e verificação de `expires_at` total. Frequência sugerida: a cada 60 segundos para timeout de rodada, a cada 5 minutos para os demais.
- **Concorrência:** todo acesso a `listings.status` deve usar `FOR UPDATE` ou equivalente para evitar race conditions entre interessados simultâneos.
- **Idempotência no Transaction Agent:** toda chamada ao Asaas (cobrança, transferência, estorno) deve ser idempotente (usar idempotency keys), já que falhas de rede podem causar retries.
- **Conciliação financeira:** rodar periodicamente a comparação entre `saldo_retido_total` (view) e o saldo real na conta Asaas da MAISOR CAPITAL. Divergência é bug crítico, não cosmético — deve gerar alerta imediato.
- **Auditoria:** `proxy_negotiation_rounds`, `negotiation_offers` e `delivery_confirmations` devem ser append-only quando possível — preferir inserir novos registros a editar/deletar histórico. Isso garante rastro completo para eventual disputa ou auditoria.
- **KYC:** nenhum usuário final (vendedor, comprador, entregador) passa por KYC bancário — apenas a conta principal da MAISOR CAPITAL tem KYC no Asaas. Validação de CPF é só de formato, não documental.
- **LGPD:** revisão jurídica formal recomendada antes do lançamento em produção, dado o volume de dados pessoais coletados (CPF, endereço, chave Pix).
- **Limites configuráveis** (sugestão de variáveis de ambiente ou tabela de configuração):
  - `MAX_RODADAS_PROXY` (default: 6)
  - `MAX_TENTATIVAS_HUMANAS` (default: 3)
  - `AJUSTE_MAXIMO_PERMITIDO` (default: -0.15)
  - `TIMEOUT_RODADA_MINUTOS` (default: 30)
  - `EXPIRACAO_TOTAL_HORAS` (default: 24)
  - `PRAZO_ESTORNO_POS_FALHA_DIAS` (default: 3)

## 10. Resumo de responsabilidades (tabela de referência rápida)

| Componente | Tipo | Quando roda | Decide dinheiro? |
|---|---|---|---|
| Conversation Agent | LLM | A cada mensagem | Não |
| Pricing/Appraisal Agent | LLM + busca + visão | Uma vez, no cadastro | Não (sugere, vendedor confirma) |
| Negotiation Engine | Código determinístico | Modo direto, a cada oferta | Sim |
| Contextual Evaluator | LLM com teto fixo | Casos-limite no modo direto | Sugere, código valida |
| Buyer/Seller Proxy Agent | LLM com guard rails | Modo proxy, entre rodadas internas | Propõe, código valida contra limite |
| Logistics/Delivery Agent | LLM + Delivery Proxy | Após acordo confirmado, antes da liberação | Propõe valor de entrega, código valida |
| Transaction Agent | Código determinístico | Após acordo confirmado / após confirmação de entrega | Executa (via Asaas, chave Pix) |
| Orquestrador | Código fino | Toda mensagem entrante | Não |
| State Store | PostgreSQL | Sempre | N/A (memória) |
